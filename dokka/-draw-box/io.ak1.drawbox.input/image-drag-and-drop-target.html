imageDragAndDropTarget

@Composable
actual fun Modifier.imageDragAndDropTarget(onImagesDropped: (drops: List<DroppedImage>) -> Unit): Modifier

No-op on Android — touch platforms don't expose OS-level file drag-drop the same way Desktop does. The existing image picker covers this flow.

@Composable
expect fun Modifier.imageDragAndDropTarget(onImagesDropped: (drops: List<DroppedImage>) -> Unit): Modifier

Attach an OS-level drag-and-drop target to a composable so files dropped onto it from Finder / Explorer / a browser tab get decoded and surfaced through onImagesDropped.

One callback fire per drop event — if the user drops three files at once, onImagesDropped receives a list of three triples in drop order:

(bytes, intrinsicSize, dropPositionScreen) for each accepted file
Content copied to clipboard

The host is expected to translate dropPositionScreen to world space via the viewport and dispatch Intent.InsertImage per entry, offsetting subsequent placements so they don't perfectly overlap.

Per-platform behavior:

  • JVM (Desktop): Hooks Modifier.dragAndDropTarget and reads event.awtTransferable.getTransferData(DataFlavor.javaFileListFlavor). Image MIME sniffing via io.ak1.drawbox.decodeImageBitmap isn't run — we just attempt ImageIO.read and skip files that can't be decoded.

  • Web (WasmJS / JS): Out of scope for this PR. The Compose Multiplatform drag-drop API doesn't surface a stable web payload yet; tracked separately.

  • Android / iOS: No-op. Touch platforms don't expose OS-level drag-drop in the same shape.
